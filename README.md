## 1. רקע ומטרות

### הבעיה הנוכחית
מסך המשאבים הנוכחי מכיל מבנה ניווט של שלוש רמות (קטגוריות → תת-קטגוריות → פריטים) שמבלבל בין נכסים מוסדיים לנכסים משויכים לעובד. השדות בכרטיס הפריט זהים לכל סוג נכס ואינם מותאמים לאופי הנכס. טפסי החתימה מגולמים בקובץ אחד של 502 שורות עם שלושה נתיבי ביצוע שזורים יחד, וכוללים באג שבו הנכס עובר לסטטוס "בשימוש" לפני שהעובד חתם בפועל.

### מטרות האפיון
- ארגון מחדש של מסך המשאבים ל-6 דומיינים עסקיים ברורים
- ניווט שטוח: דומיין → פריט (שתי רמות בלבד)
- שיוך מהיר של מספר נכסים לעובד בפעולה אחת
- החלפת טופס החתימה הגנרי ב-5 פרוטוקולים ממוקדים לפי סוג נכס
- מודול Offboarding אוטומטי עם צ'קליסט ודוח מצב שוטף

## 2. ארכיטקטורת מסך משאבים

### מבנה ניווט חדש

```
/assets                          ← מסך ראשי: 6 כרטיסי דומיין
/assets/:domain                  ← רשימת פריטים בדומיין
/assets/:domain/:itemId          ← פרטי פריט
```

הניווט הנוכחי (קטגוריות → תת-קטגוריות → פריטים) **מבוטל**. הסוג (למשל "MacBook Pro") הופך לשדה על הפריט עצמו, לא לרמת תפריט.

### מסך ראשי — מבנה

**שורת סיכום עליונה (dashboard bar):**
- מספר פריטים פגי תוקף (אדום)
- מספר פריטים הפגים תוך 30 יום (כתום)
- מספר טפסי offboarding פתוחים
- לחיצה על כל נתון → רשימה מסוננת

**רשת 6 דומיינים:**
כרטיס לכל דומיין עם: אייקון, שם, תיאור קצר, מספר פריטים, אינדיקטור תפוגה אם רלוונטי.

**כפתור "שיוך מהיר לעובד"** בכותרת המסך — כניסה לזרימת שיוך מהיר (ראה סעיף 4).

**חיפוש גלובלי** — מחפש בכל הדומיינים, מחזיר תוצאות עם ציון הדומיין שלהן.

### 3.1 ציוד פיזי

**תצוגת רשימה:**
| עמודה | הערות |
|-------|-------|

מיון ברירת מחדל: לפי תאריך עדכון אחרון

**פרטי פריט — שדות:**
- קטגוריה, שם, קוד (אוטומטי), יצרן/דגם, מס' סידורי

- תאריך רכישה, ערך כספי
- תאריך תפוגת אחריות

**רכב חברה — שדות נוספים:**
- מספר רישוי, דגם, שנת ייצור
- קילומטראז' נוכחי (שדה עדכני)
- תאריך טיפול אחרון / הבא

### 3.3 רשיונות ותוכנות

**תיאור:** מנויי SaaS לעובד (Slack, Zoom, Adobe), רשיונות חברה (Microsoft 365), מנויים לשירותי ענן.

**תצוגת רשימה — תצוגת כרטיסים (לא טבלה):**
כרטיס לכל תוכנה עם: לוגו/אייקון, שם, סה"כ מושבים, בשימוש, פנויים, עלות חודשית, תאריך חידוש.

לחיצה על כרטיס → רשימת עובדים עם מושב בתוכנה זו.

**סוגי רשיון:**
- `מושב אישי לעובד` — משויך לעובד ספציפי
- `רשיון חברה` — מוסדי, לא לעובד ספציפי
- `concurrent` — X משתמשים בו-זמנית

**פרטי פריט — שדות:**
- שם תוכנה, ספק, סוג רשיון
- כמות מושבים רכושה / בשימוש / פנויים
- עלות לחודש / לשנה
- תאריך חידוש (עם התראה 30 יום מראש)
- איש קשר בספק + טלפון
- קישור לניהול מנוי
- רשימת עובדים עם מושב
- מסמכים (חוזה, חשבונית)

### 3.5 הדרכות ותאימות

**תיאור:** הדרכות בטיחות, הטרדה מינית, נגישות, אישורים תקופתיים — כל הדרכה עם תוקף.

**תצוגת רשימה — מטריצת עובד × הדרכה:**
שורות = עובדים, עמודות = הדרכות, תאים = סטטוס עם צביעה.
- ירוק: הושלם ובתוקף
- כתום: יפוג תוך 60 יום
- אדום: פג תוקף / לא בוצע

**חלופה:** תצוגת טבלה פשוטה עם פילטר לפי הדרכה או לפי עובד.

**פרטי הדרכה — שדות:**
- שם הדרכה
- גוף מנחה / מרצה
- תדירות חידוש: חד-פעמי / חצי שנה / שנה / שנתיים
- רשימת עובדים החייבים בהדרכה (לפי תפקיד/מחלקה)
- לכל עובד: סטטוס, תאריך השלמה, תאריך תפוגה, קישור ל-PDF אישור

### 3.6 נדל"ן וחוזים

דומיין זה מתחלק לשני תת-דומיינים עם לוגיקה שונה לחלוטין.

#### 3.6.1 נכסים מניבים (הכנסה)
נכסים שהחברה מחזיקה ומשכירה לאחרים.

**תצוגת רשימה:**
| עמודה | הערות |
|-------|-------|
| כתובת | |
| שוכר | |
| שכ"ד חודשי | |
| תחילה / סיום | |
| ימים לסיום | אדום אם < 90 יום |
| סטטוס | חוזה פעיל / מסתיים / פנוי / סכסוך |

**פרטי פריט — שדות:**
- כתובת מלאה, גוש/חלקה/תת-חלקה
- סוג נכס (משרד, מסחרי, מחסן, דירה)
- שטח ברוטו/נטו (מ"ר), קומה, כניסה, מספר חניות
- מספר נכס פנימי (קוד)
- שם שוכר, ח.פ./ת.ז., איש קשר, טלפון, מייל, כתובת למשלוח דואר
- תאריך תחילה/סיום חוזה
- אופציית הארכה: כן/לא + תנאים + תאריך הודעה
- דמי שכירות חודשיים
- מנגנון הצמדה: מדד / דולר / קבוע
- מועד תשלום (יום בחודש)
- תשלום ועד בית + ארנונה: מי משלם
- ערבויות: ערבות בנקאית (בנק, סכום, תוקף), שיק ביטחון (סכום, מספר), ערב אישי (שם, ת.ז.), פיקדון מזומן
- מסמכים: חוזה שכירות, נסח טאבו, תשריט, נספחים
- היסטוריית תשלומים (לשונית נפרדת): חודש, סכום, תאריך קבלה, סטטוס (שולם/באיחור/חסר)

**ניהול חיצוני (Toggle — כבוי כברירת מחדל):**
כאשר "הנכס מנוהל ע"י חברה חיצונית" מופעל, מופיעים שדות נוספים:
- שם חברת הניהול, ח.פ., איש קשר, טלפון, מייל
- תאריך תחילת הסכם ניהול
- אחוז עמלה מהשכ"ד / עמלה קבועה חודשית
- מי משלם את העמלה: חברה / שוכר
- מסמך הסכם ניהול

כשהניהול מופעל: התראות תפוגה נשלחות גם לאיש הקשר בחברת הניהול. עמלת הניהול מחושבת ומוצגת כ"עלות נטו" בסיכום הנכס.

## 4. שיוך מהיר לעובד חדש

### נקודות כניסה
1. **ממסך עובדים** — לאחר הקמת עובד חדש מופיע כפתור "שייך משאבים". לחיצה → מסך שיוך מהיר עם העובד כבר ממולא.
2. **ממסך משאבים** — כפתור "שיוך מהיר לעובד" בכותרת → dropdown לבחירת עובד → מסך שיוך.

### מבנה מסך השיוך

**כותרת:** שם העובד + תפקיד + אפשרות "שינוי עובד"

**טאבים לפי דומיין:**
- ציוד פיזי (מציג רק פריטים בסטטוס "במלאי")
- גישות דיגיטליות (מציג רשימת סוגי גישות שהחברה מעניקה)
- רשיונות ותוכנות (מציג תוכנות עם מושבים פנויים)
- הדרכות (מציג הדרכות שהעובד עדיין לא ביצע)

בכל טאב — ספירת פריטים מסומנים בתגית.

**טבלת פריטים (בתוך כל טאב):**
- שדה חיפוש
- קיבוץ לפי קטגוריה עם כותרת קבוצה
- checkbox לכל שורה, select-all לקבוצה
- עמודות: שם, קוד, סטטוס נוכחי, שדה הערה אופציונלי

**שורת "נכסים חדשים" (בטאב ציוד פיזי בלבד):**
- checkbox בצבע שונה (סגול) להבחנה ויזואלית
- שדות: שם נכס + פרטים מינימליים (דגם, סריאלי)
- הנכס ייצור **וישויך** בו-זמנית
- לאחר שיוך — ניתן להשלים פרטים בפרטי הפריט

**שורת הסיכום בתחתית:**
- `X נכסים קיימים + Y חדשים לשיוך`
- כפתורות: ביטול | שייך ← N נכסים

**גישות דיגיטליות — התנהגות שונה:**
בטאב גישות אין "מלאי". מוצגת רשימת סוגי גישות שהחברה מגדירה. בסימון — נוצרת רשומת גישה חדשה עבור העובד.

**לאחר שיוך:**
אם יש נכסים שדורשים חתימה → הצעה לפתוח פרוטוקול חתימה. ניתן לדחות ולבצע מאוחר יותר.

## 5. פרוטוקולי חתימה

### עקרון כללי
מחליף את `AssignAssetWithFormDialog.tsx` הנוכחי. **אין שליחה לפורטל.** העובד חותם ישירות מול מנהל התפעול על המכשיר, או מועלה מסמך סרוק.

### שני מסלולי ביצוע

**מסלול A — חתימה דיגיטלית:**
מסך מלא (לא דיאלוג). תצוגת הטופס בצד שמאל, לוח חתימה גדול (מינימום 400px) בצד ימין. לחיצה על "שמור חתימה" → PDF נוצר → נשמר ב-Supabase Storage → סטטוסים מתעדכנים.

**מסלול B — העלאת סריקה:**
העלאת PDF או תמונה. תצוגה מקדימה לפני אישור. ללא לוח חתימה. המסמך הסרוק = הקובץ הרשמי. ישר לאישור ושמירה.

### זיהוי פרוטוקול אוטומטי
המערכת מזהה איזה פרוטוקול להפעיל לפי `asset_categories.protocol_type`:
- `physical` → פרוטוקול 1
- `virtual` → פרוטוקול 2

## 6. מודול Offboarding

### פתיחת תהליך

**נקודת כניסה:** שינוי סטטוס עובד ל-`leaving` במסך עובדים → מופיע כפתור "פתח תהליך Offboarding".

**בפתיחה — שדות חובה:**
- תאריך אחרון בעבודה

**יצירת צ'קליסט אוטומטי:**
DB function סורקת ויוצרת שורה ב-`offboarding_items` לכל:
- נכס פיזי עם `current_owner_id = employee_id`
- גישה דיגיטלית עם `status = active` ו-`employee_id`
- מושב תוכנה של העובד
- הדרכות שלא בוצעו (אם רלוונטי)

### לוח הבדיקות

**מבנה:** כרטיס עם שם עובד, תפקיד, תאריך עזיבה, שורת progress.

**סעיפים:**

**1. ציוד פיזי להחזרה**
- רשימת פריטים עם: שם, קוד, כפתור "בצע החזרה" → פותח פרוטוקול 5א
- לאחר ביצוע → checkbox מתמלא, תאריך החזרה מוצג

**2. גישות דיגיטליות לניתוק**
- רשימת גישות עם: שם מערכת, סוג, כפתור "בצע ניתוק" → פותח פרוטוקול 5ב
- אחראי: איש IT

**3. רשיונות תוכנה לשחרור**
- רשימת מושבים אישיים עם כפתור "שחרר מושב"
- שחרור מושב = ביטול השיוך בדומיין רשיונות ותוכנות
- אחראי: איש IT / מנהל תפעול

**4. מסמכים וחתימות**
- טופס ניתוק גישות (IT sign-off) — כפתור "צור טופס"
- אישור סיום העסקה (HR) — כפתור "צור טופס"

### בעלות משימות

| סוג משימה | אחראי ראשי | מקבל התראה | מסמן הושלם |
|-----------|-----------|------------|------------|
| החזרת ציוד פיזי | מנהל תפעול | אדמין | מנהל תפעול |
| ניתוק גישות | איש IT | אדמין | איש IT |
| שחרור מושבי תוכנה | איש IT | מנהל תפעול | איש IT |
| טפסי חתימה ומסמכים | מנהל תפעול | אדמין | מנהל תפעול |
| אישור סגירת תהליך | אדמין | IT + תפעול | אדמין |

### ייצוא דו"ח

**עיקרון:** הדו"ח ניתן לייצוא **בכל עת** ומייצג את המצב הנוכחי ברגע הייצוא — לא רק בסיום.

**תוכן הדו"ח (PDF):**
- שם עובד, תפקיד, מחלקה, תאריך תחילת העסקה, תאריך עזיבה
- סעיפים שהושלמו: מה, מתי, על ידי מי + קישור ל-PDF טופס
- סעיפים פתוחים: פירוט, שם אחראי, תפקיד, ימים עד עזיבה
- חתימות: עובד, מנהל ישיר, HR, IT
- חותמת זמן הפקה

**הערה:** ניתן לסמן פריטים כ-`not_applicable` עם הסבר. פריטים כאלו מוצגים בדו"ח עם הסיבה — לא חוסמים ייצוא.

## 8. שינויי DB נדרשים

### 8.1 טבלאות חדשות

#### `signed_documents` — מחליפה את `asset_handover_forms`
```sql
create table signed_documents (
  id              uuid primary key default gen_random_uuid(),
  company_id      uuid not null references companies(id),
  protocol_type   text not null, -- physical | virtual | vehicle | training | return_physical | return_virtual
  employee_id     uuid references employees(id),
  asset_id        uuid references assets(id),
  training_id     uuid references trainings(id),
  issued_at       timestamptz not null default now(),
  returned_at     timestamptz,              -- מולא בפרוטוקול 5
  method          text not null,            -- digital | scan
  form_snapshot   jsonb,                    -- snapshot של הנתונים בעת החתימה
  employee_signature text,                  -- data URL
  issuer_signature   text,
  pdf_url         text,
  attached_doc_url text,
  signed_by       uuid references auth.users(id),
  notes           text,
  created_at      timestamptz default now()
);
```

**Migration:** העתקת רשומות מ-`asset_handover_forms` עם `protocol_type = 'physical'`.

---

#### `offboarding_processes`
```sql
create table offboarding_processes (
  id                uuid primary key default gen_random_uuid(),
  company_id        uuid not null references companies(id),
  employee_id       uuid not null references employees(id),
  last_working_day  date not null,
  status            text not null default 'in_progress', -- in_progress | completed
  completed_at      timestamptz,
  pdf_url           text,
  created_by        uuid references auth.users(id),
  created_at        timestamptz default now()
);
```

---

#### `offboarding_items`
```sql
create table offboarding_items (
  id              uuid primary key default gen_random_uuid(),
  process_id      uuid not null references offboarding_processes(id) on delete cascade,
  item_type       text not null,   -- asset | access | license | document
  ref_id          uuid,            -- asset_id / access_id / license_id
  title           text not null,
  owner_role      text not null,   -- it | ops | admin
  status          text not null default 'pending', -- pending | done | not_applicable
  completed_at    timestamptz,
  completed_by    uuid references auth.users(id),
  notes           text
);
```
